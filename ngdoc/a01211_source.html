31 /**
32  * \addtogroup sensortag-cc26xx-peripherals
33  * @{
34  *
35  * \defgroup sensortag-cc26xx-hdc-sensor SensorTag 2.0 TI HDC1000 Sensor
36  *
37  * Due to the time required for the sensor to startup, this driver is meant to
38  * be used in an asynchronous fashion. The caller must first activate the
39  * sensor by calling SENSORS_ACTIVATE(). This will trigger the sensor's startup
40  * sequence, but the call will not wait for it to complete so that the CPU can
41  * perform other tasks or drop to a low power mode. Once the sensor has taken
42  * readings, it will automatically go back to low power mode.
43  *
44  * Once the sensor is stable, the driver will retrieve readings from the sensor
45  * and latch them. It will then generate a sensors_changed event.
46  *
47  * The user can then retrieve readings by calling .value() and by passing
48  * either HDC_1000_SENSOR_TYPE_TEMP or HDC_1000_SENSOR_TYPE_HUMIDITY as the
49  * argument. Multiple calls to value() will not trigger new readings, they will
50  * simply return the most recent latched values.
51  *
52  * The user can query the sensor's status by calling .status()
53  *
54  * To get a fresh reading, the user must trigger a new reading cycle by calling
55  * SENSORS_ACTIVATE().
56  * @{
57  *
58  * \file
59  * Header file for the Sensortag TI HDC1000 sensor
60  */
